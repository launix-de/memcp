## Branch Purpose

This branch (`jit-tailcall`) fixes register-allocation correctness bugs in the JIT
compiler for storage types (StorageSeq, StorageInt, etc.).  The symptom is wrong
query results when JIT-compiled scan functions are used with multiple sequences
(e.g. `TestStorageSeqJITEmitMultipleSequences` returning wrong values for idx â‰Ą 5).

## Key Architecture: JIT Emitter-Generator

### How it works (3 layers)

1. **`tools/jitgen/main.go`** â€” code generator (Go SSA â†’ Go code)
   - Reads storage Go source files, builds SSA, generates `JITEmit` methods.
   - Run with: `make jitgen`
   - Patches generated code into `storage/storage-*.go` files between markers.
   - **THIS IS THE FILE TO FIX** when register-scheduler bugs are found.
   - Do NOT manually edit the generated sections in `storage/storage-*.go`.

2. **`storage/storage-seq.go`** (and `storage-int.go`, etc.) â€” auto-generated
   - Contains `JITEmit` methods generated by jitgen.
   - Regenerated by `make jitgen`. Manual patches are overwritten.

3. **`scm/jit_types.go`** â€” runtime register allocator
   - `JITContext.AllocReg()`: picks from `FreeRegs &^ ProtectedRegs`; if none
     available, **evicts** the highest non-protected register (spills to SpillBuf,
     increments `EvictedCounts[r]`, returns `r` as fresh).
   - `FreeReg(r)`: if `EvictedCounts[r] > 0`, burns the token; else returns to FreeRegs.
   - `ProtectReg(r)` / `UnprotectReg(r)`: reference-counted protection bitmask.
   - `TransferReg(r)`: burns eviction token when result aliases input register.

4. **`scm/jit_emit_amd64.go`** â€” AMD64 machine code emission helpers
   - `EmitMovRegReg`, `EmitShrRegImm8`, `EmitSetcc`, `EmitCmpRegImm32`, etc.
   - All register arguments are `uint8` (abstract register numbers).

## The Core Bug Class: Eviction-Alias

When `AllocReg()` is called while a value `src` is live in a register and all
registers are busy, the allocator evicts `src.Reg` (spills to memory) and returns
**the same register** as the new allocation target.  The generated code then does:

```
dst := ctx.AllocReg()            // dst == src.Reg (evicted!)
ctx.W.EmitMovRegReg(dst, src.Reg)  // self-mov, no-op
ctx.W.EmitShrRegImm8(dst, 6)       // src.Reg now holds wrong value
```

Or for comparisons:
```
rv := ctx.AllocReg()             // rv == src.Reg (evicted!)
ctx.W.EmitCmpRegImm32(src.Reg, imm32)  // CMP reads src.Reg OK (still original value)
ctx.W.EmitSetcc(rv, CcEQ)        // SetCC writes 0/1 to rv == src.Reg â€” corrupts src!
```

**Fix pattern** (already in the slice-load code at ~line 1563):
```go
ctx.ProtectReg(src.Reg)   // prevent eviction
dst := ctx.AllocReg()     // guaranteed dst != src.Reg
ctx.UnprotectReg(src.Reg) // allow eviction again
ctx.W.EmitMovRegReg(dst, src.Reg)  // correct copy
```

## What the jitgen fix does

In `tools/jitgen/main.go`, the `emitAllocRegProtected` helper was added:

```go
func (g *codeGen) emitAllocRegProtected(dstVar, indent string, guard bool, gv genVal)
```

It emits `ProtectReg` + `AllocReg` + `UnprotectReg` when `guard == true`.
It is called with `guard = xMultiUse` (true when the source value is used again
after the current instruction).

The `xMultiUse` flag at each BinOp site:
```go
xMultiUse = g.ssaValueUsesRemaining(v.X.Name()) > 1
```

This flag drives whether to copy-before-modify (to preserve src).  When true, the
copy `AllocReg()` must also be wrapped with protect/unprotect.

### Affected BinOp patterns in jitgen (all now use emitAllocRegProtected):
- Comparison (`cc != ""`): const-Y, y-imm, both-reg paths
- Arithmetic ADD/SUB/MUL: const-Y (SUB / ADD|MUL), identity y=0, y-imm, both-reg
- Division `QUO`: const divisor
- Modulo `REM`: const divisor
- Shift `SHL`/`SHR`: const amount, y-imm, variable (double-indent)

## Register naming convention

- Generated code uses abstract register numbers (`uint8`), NOT platform register names.
- Register names (`RegRAX`, `RegR15`, etc.) appear only in `scm/jit_emit_amd64.go`.
- **Never hardcode register names** (like `r15`, `r12`) in non-`_amd64` files.
- The register scheduler assigns concrete registers at runtime; the generator must
  only emit logical protect/allocate/free calls.
